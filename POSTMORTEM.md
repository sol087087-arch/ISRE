# ISRE — Постмортем: где обосрались

Честный разбор ошибок по ходу проекта. Без прикрас. Цель — не самобичевание,
а чтобы те же грабли не прилетели в лоб второй раз.

Дата составления: 2026-05-16. Автор разбора: рабочая сессия с Claude.

---

## TL;DR

Прототип был написан, но **ни разу не прогонялся end-to-end с проверкой
ground-truth**. Из-за этого четыре класса ошибок копились молча, и каждый
всплывал только когда предыдущий чинился. Хронология провалов:

1. **Метрика успеха была фейковой** — `_is_canonical` = "движок не знает ходов",
   а не "это канон". Завышала всё.
2. **Движок недогенерировал** — 7 багов, 48% ENGINE_GAP на random.
3. **Датасет учил 9 действий из 10** — SORT_COMMUTATIVE отсутствовал (0%).
4. **Датасет содержал циклы** — 12.1% траекторий с повторяющимся состоянием.
5. **Чинили движок поверх уже сгенерированных данных** — дважды пришлось
   перегенерировать 50K, потому что фиксы движка ломали консистентность данных.

Каждый из этих пунктов делал предыдущие измерения невалидными. Суммарно
выброшено: 1 MLP-run на ~13 GPU-часов + 2 полных датасета по 50K + первые
baseline-числа.

---

## Провал №1: фейковая метрика канона

### Что сделали не так

`_is_canonical(state)` был реализован как `len(get_candidates(state)) == 0`.
Логика: "если движок не предлагает ходов — значит дошли до канона".

Это **конфляция двух разных вещей**:
- честный канон (дальше упрощать нечего)
- тупик движка (движок не знает что делать, хотя упрощение существует)

### Чем аукнулось

Первые baseline-числа: random 98.6%, greedy 100%. Выглядело прекрасно.
После замены на ground-truth (`root.to_expr() == canonical.to_expr()`):
random упал до 47.2%, greedy до 92.5%.

**48% "успехов" random были фейком.** Половина отчётных цифр — мусор.

### Корневая причина

Метрика успеха никогда не проверялась против заранее известного ответа.
`canonical_ast` лежал в каждой траектории с самого начала — его просто
не использовали для сравнения.

### Урок

Метрику успеха валидировать против ground-truth С ПЕРВОГО ДНЯ. Если в данных
есть правильный ответ — сравнивать с ним, а не с косвенным признаком
("движок замолчал" ≠ "задача решена").

## Провал №2: 7 багов в символьном движке

### Что сделали не так

Движок написан, у каждого действия есть `__main__`-тест на match+apply.
Тесты проходили. Но тесты проверяли отдельные действия на игрушечных входах,
а не покрытие на реальном распределении.

7 багов:
1. `_can_fold_const`: требовал ALL children = NUMBER
2. `_sort_key`: клал константы ПЕРВЫМИ в ADD (канон — последними)
3. один sort_key на ADD и MUL (разный канон)
4. `_can_remove_zero`: только ADD, не MUL
5. `_can_remove_one`: не покрывал Pow(x,1)
6. `_can_merge_power`: не видел VARIABLE как x^1
7. `_can_fold_const` MUL: требовал ALL = NUMBER

### Чем аукнулось

48% ENGINE_GAP на random, 8% на greedy. Движок возвращал пустой список
кандидатов для состояний, которые НЕ канон. Это контаминировало обучающие
данные: gold-действия генерировались движком, баги движка → кривые данные.

### Корневая причина

Юнит-тесты на действия ≠ покрытие на распределении. Баг #2 (sort order)
вообще не ловился ни одним тестом — потому что тест проверял "SORT_COMMUTATIVE
применяется к Add(x,1)", а это уже было закодировано под СТАРЫЙ неправильный
порядок. Тест подтверждал багованное поведение.

### Урок

После написания движка — диагностический прогон на N случайных примерах
из реального распределения с классификацией провалов (ENGINE_GAP / LOOP /
MAX_STEPS). `diagnose_failures.py` надо было писать ДО первых baseline-чисел,
а не после.

## Провал №3: датасет учил 9 действий из 10

### Что сделали не так

После фикса 7 багов движка перегенерировали 50K. Проверили ENGINE_GAP = 0%.
Обрадовались. Запустили MLP-тренировку.

**Не проверили распределение gold-действий перед тренировкой.**
SORT_COMMUTATIVE = 0% из 150K обучающих пар. В INVERSE_REGISTRY не было
обратной операции, которая бы генерировала состояния, требующие
SORT_COMMUTATIVE как gold-действие.

### Чем аукнулось

MLP-run (эпохи 1–6, val_acc 74%) обучен на данных, где одного из 10 типов
действий нет вообще. Run полностью невалиден. ~несколько GPU-часов в мусор.

### Корневая причина

Перепутали "движок умеет действие" с "генератор создаёт примеры этого
действия как gold". Движок SORT_COMMUTATIVE поддерживал. Генератор —
никогда не создавал состояний, где он gold. Это разные подсистемы,
проверять надо обе.

### Урок

`analyze_dataset.py` (распределение gold-действий) — ОБЯЗАТЕЛЬНЫЙ гейт
перед любой тренировкой. Все 10 действий должны присутствовать с
ненулевой долей. Это одна команда, 30 секунд. Пропустили — потеряли run.

## Провал №4: 12.1% траекторий с циклами

### Что сделали не так

Добавили SHUFFLE_COMMUTATIVE inverse → SORT_COMMUTATIVE появился (14%).
Перегенерировали 50K (v3). Запустили analyze_dataset — **только тогда**
заметили: 12.1% траекторий содержат повторяющееся состояние (A→B→A).

### Чем аукнулось

Цикл в траектории = противоречивый обучающий сигнал: модель видит одно
и то же состояние с разными gold-действиями по структурно бессмысленной
причине. Датасет v3 пришлось выбросить и чинить генератор.

### Корневая причина

SHUFFLE_COMMUTATIVE взаимодействовал с движком тремя способами, каждый
давал цикл:
- INV_NOOP: shuffle попадал в тот же порядок → состояние не менялось
- FWD_NOOP: SORT no-op на детях с равными sort-key
- BACK_CYCLE: SORT возвращал к более раннему состоянию цепочки

Ни один не был очевиден из дизайна. Всплыли только на диагностике 50K.
Плюс первый фикс (`chain_exprs`) был слишком агрессивным — отсёк 80%
валидных траекторий (Failed 2412/3000), потому что `_after == current`
для корректных шагов это норма, а я проверял против всей цепочки включая
current. Чинили фикс фикса.

### Урок

Новая inverse-операция = обязательная проверка на циклы на маленькой
выборке (2–3K) ПЕРЕД полным датасетом. Цена пропуска — 35 минут
регенерации + выброшенный анализ.

## Провал №5: чинили движок поверх готовых данных (дважды)

### Что сделали не так

Паттерн повторился буквально:
1. Сгенерировали 50K.
2. Прилетела критика по коду движка.
3. Пофиксили движок.
4. Осознали, что фиксы меняют поведение `engine.apply` →
   уже сгенерированный датасет несовместим → регенерация.

Так было с v2→v3 (фикс sort_key, добавление SHUFFLE) и v4→v5
(фикс combine_coeff symmetry + coeff order из внешней критики).

### Чем аукнулось

- v2: выброшен (0% SORT_COMMUTATIVE)
- v3: выброшен (12.1% циклов)
- v4: выброшен (несовместим с исправленным combine_coeff/coeff_key)
- MLP-run на v4 (~13 GPU-часов, 20 эпох) — диагностика, не результат

Итого 3 полных датасета и 1 длинный train-run в мусор.

### Корневая причина

Движок и данные — связанная пара. Любое изменение `get_candidates` или
`apply` инвалидирует все ранее сгенерированные траектории для
rollout-метрик. Это не было осознано как жёсткий инвариант: данные
генерировались, ПОТОМ движок ревьюился и правился.

### Урок

**Заморозить движок перед генерацией финального датасета.** Code review
движка — ДО генерации, не после. Если движок меняется — данные
автоматически считаются невалидными для rollout, без исключений.

Конкретный гейт: тег `engine-frozen-vN` в git. Датасет генерируется
только с замороженного тега. Любой коммит в `symbolic_engine.py` после
заморозки → новый тег → обязательная регенерация.

## Что сделано правильно (чтобы не только посыпать голову пеплом)

- Ground-truth сравнение в итоге внедрено — фундамент честных метрик.
- `diagnose_failures.py` с таксономией провалов — поймал ENGINE_GAP
  систематически, а не на глаз.
- `inverse_skip_counts` диагностика — рентген генератора, видно какие
  inverse недогенерируют.
- Внешнюю критику кода проверяли по фактическому коду, а не на веру —
  пункт 4 оказался не багом, пункт 6 не багом сейчас. Не чинили то,
  что не сломано.
- Каждый фикс верифицировался на маленькой выборке перед полным прогоном
  (после провала №4 это стало правилом).
